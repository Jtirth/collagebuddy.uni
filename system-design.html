System Design Fundamentals

Learn to design scalable and reliable systems

Scalability Beginner

Scalability is the ability of a system to handle increased load by adding resources. Learn about vertical and horizontal scaling strategies.

Key Concepts:

  • Vertical Scaling (Scale Up): Adding more power (CPU, RAM) to existing machines
  • Horizontal Scaling (Scale Out): Adding more machines to distribute the load
  • Load Balancing: Distributing traffic across multiple servers
  • Caching: Storing frequently accessed data in memory for faster retrieval

Real-World Example:

  • Netflix uses horizontal scaling to handle millions of concurrent users
  • They distribute content across multiple servers worldwide (CDN)
  • Load balancers route user requests to the nearest available server

Best Practices:

  • Design for horizontal scaling from the start
  • Use stateless services when possible
  • Implement caching at multiple levels (browser, CDN, application, database)
  • Monitor system metrics to identify bottlenecks
Database Design Beginner

Learn how to design efficient databases, choose between SQL and NoSQL, and implement proper indexing and partitioning strategies.

SQL vs NoSQL:

  • SQL (Relational): Structured data, ACID compliance, complex queries (MySQL, PostgreSQL)
  • NoSQL: Flexible schema, horizontal scaling, high performance (MongoDB, Cassandra)

Database Optimization:

  • Indexing: Create indexes on frequently queried columns
  • Normalization: Organize data to reduce redundancy
  • Denormalization: Sometimes duplicate data for faster reads
  • Partitioning/Sharding: Split large databases across multiple servers

Example Use Cases:

  • E-commerce: Use SQL for transactions, NoSQL for product catalog
  • Social Media: NoSQL for user posts, SQL for user authentication
  • Analytics: Data warehouses like Redshift for complex queries
Caching Strategies Intermediate

Caching improves performance by storing frequently accessed data in fast storage. Learn about different caching layers and strategies.

Caching Layers:

  • Browser Cache: Stores static assets (images, CSS, JS) locally
  • CDN Cache: Distributes content geographically (Cloudflare, AWS CloudFront)
  • Application Cache: In-memory caching (Redis, Memcached)
  • Database Cache: Query result caching

Cache Invalidation Strategies:

  • Time-based (TTL): Cache expires after a set time
  • Event-based: Invalidate cache when data changes
  • LRU (Least Recently Used): Remove least accessed items when cache is full

Common Patterns:

  • Cache-Aside: Application checks cache first, then database
  • Write-Through: Write to cache and database simultaneously
  • Write-Behind: Write to cache first, database later (async)
Load Balancing Intermediate

Distribute incoming traffic across multiple servers to ensure no single server is overwhelmed and improve reliability.

Load Balancing Algorithms:

  • Round Robin: Distribute requests equally across all servers
  • Least Connections: Send to server with fewest active connections
  • IP Hash: Route based on client IP address (session persistence)
  • Weighted Round Robin: Distribute based on server capacity

Types of Load Balancers:

  • Layer 4 (Transport): Routes based on IP and port (faster, simpler)
  • Layer 7 (Application): Routes based on content (HTTP headers, cookies)

Popular Load Balancers:

  • NGINX, HAProxy (Software)
  • AWS ELB, Google Cloud Load Balancing (Cloud)
  • Hardware load balancers (F5, Citrix)
Microservices Architecture Advanced

Break down monolithic applications into smaller, independent services that can be developed, deployed, and scaled independently.

Key Principles:

  • Single Responsibility: Each service does one thing well
  • Decentralized Data: Each service manages its own database
  • Independent Deployment: Services can be updated without affecting others
  • API Gateway: Single entry point for all client requests

Communication Patterns:

  • Synchronous: REST APIs, gRPC (request-response)
  • Asynchronous: Message queues (RabbitMQ, Kafka)
  • Event-Driven: Services publish/subscribe to events

Challenges:

  • Distributed system complexity
  • Service discovery and orchestration (Kubernetes)
  • Data consistency across services
  • Monitoring and debugging

When to Use:

  • Large teams working on different features
  • Need to scale specific components independently
  • Different services require different technologies
